# @Browser 长程任务案例与提示词模板

本文整理适合 `@Browser` 的长程智能体任务类型，并提供可直接复制到 Machi 的提示词模板。

适用目标：
- 任务步骤多，且需要跨页面连续操作；
- 需要阶段性可见进度，而不是一次性回答；
- 需要明确落盘产物（报告、清单、截图索引）。

## 一、推荐任务案例

### 1) 竞品多站点深度调研
- 场景：连续访问官网、文档、定价页、更新日志，抽取结构化信息后汇总。
- 产物：`docs/research/competitor_report.md`
- 关键验收：覆盖来源链接、信息去重、结论可追溯。

### 2) 核心业务流程回归测试
- 场景：登录 -> 配置 -> 提交 -> 结果页的完整链路验证。
- 产物：`docs/qa/e2e_smoke_result.md`
- 关键验收：每一步有通过/失败状态和复现描述。

### 3) SEO 与可访问性巡检
- 场景：批量检查页面标题、meta、死链、图片 alt、基础可访问性问题。
- 产物：`docs/audit/seo_accessibility_audit.md`
- 关键验收：问题分级（高/中/低）和修复优先级。

### 4) 后台系统批量录入与核对
- 场景：按规则批量填写表单、提交后核对结果页。
- 产物：`docs/ops/batch_form_execution_log.md`
- 关键验收：失败项单独列出并可重试。

### 5) 资讯监控与日报生成
- 场景：多来源抓取新内容，按关键词筛选并生成日报。
- 产物：`docs/reports/browser_daily_digest_YYYYMMDD.md`
- 关键验收：去重、来源可信度标注、摘要精炼。

### 6) 文档站迁移对比检查
- 场景：新旧站点逐页比对标题、段落、链接可达性。
- 产物：`docs/migration/site_diff_report.md`
- 关键验收：缺失项列表与影响范围清晰。

## 二、可复制提示词模板

### 模板 A：长程调研类

```text
@Browser
目标：对 <主题> 做多站点调研，并输出可落盘报告。

执行要求：
1. 分阶段执行：先列计划，再逐站采集，最后汇总结论。
2. 每完成一个阶段都汇报当前进度、已完成项、下一步。
3. 所有关键结论都要附来源 URL。
4. 发现冲突信息时标注“待确认”并给出判断依据。
5. 最终必须写入文件：<目标路径>.md

验收标准：
- 报告包含：结论摘要、证据链接、风险与建议、后续行动。
- 如果无法落盘，必须明确报错并给出可重试命令，不得口头宣称完成。
```

### 模板 B：流程测试类

```text
@Browser
目标：验证 <业务流程名> 从入口到结果页全链路可用。

执行要求：
1. 使用“步骤编号”输出：步骤、预期、实际、结果（PASS/FAIL）。
2. 每步失败后先重试 1 次；仍失败则继续后续步骤并记录阻断影响。
3. 输出失败复现条件和建议修复点。
4. 最终写入文件：<目标路径>.md

验收标准：
- 文件里必须有总通过率、失败明细、阻断级问题列表。
```

### 模板 C：批处理执行类

```text
@Browser
目标：批量处理 <任务描述>，并输出执行日志。

执行要求：
1. 先给出批处理计划（总条数、分批策略、失败重试策略）。
2. 每批次完成后输出成功数、失败数、失败原因分类。
3. 对失败项自动重试 1 次，并在日志中标记是否恢复成功。
4. 最终写入文件：<目标路径>.md

验收标准：
- 日志包含批次统计、失败条目、重试结果、剩余待办。
```

## 三、稳定执行建议

- 明确要求“分阶段汇报”，避免长时间无可见反馈。
- 明确“最终必须落盘”，并指定绝对路径。
- 明确“失败时不得口头完成”，必须返回错误与重试建议。
- 对高耗时任务设置阶段性里程碑，便于中断后续跑。
- 对多页面任务先做 URL 清单，再执行，避免重复访问。

## 四、建议的落盘路径规范

- 调研报告：`docs/research/*.md`
- 测试报告：`docs/qa/*.md`
- 巡检报告：`docs/audit/*.md`
- 运行日志：`docs/ops/*.md`

这样便于后续检索、引用和自动化汇总。
