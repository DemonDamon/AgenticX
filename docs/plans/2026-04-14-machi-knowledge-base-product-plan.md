# Machi 知识库能力 — 产品规划

> **目标读者：** 产品 / 架构 / 前后端工程师；落地实施时可将子阶段拆成 `.cursor/plans/` 下的实现计划并配 `Plan-Id` 追溯。

**Goal：** 为 Machi 提供可演进的 **知识库（Knowledge Base, KB）** 能力：支持 **本地构建与混合检索问答**，并 **兼容远端知识源**（含 MCP 与厂商 API），与现有会话、工作区、工具与 MCP 生态对齐，避免「只做聊天附件」与「企业级 KMS」混为一谈。

**非目标（首期明确不做或降级）：** 替代完整企业文档治理（细粒度 ACL 审计、合规留存策略）；跨设备实时协作编辑；保证所有厂商 KB 的功能对等。

---

## 1. 背景与问题陈述

用户需要在 Machi 内：

1. **把本地资料变成可检索的长期知识**（拖拽 / 选定目录 → 解析 → 入库 → 问答），而不是仅当轮 `context_files` 全文塞入提示词。
2. **连接已有远端知识库**（自建、AgenticX 生态、第三方云 KB），避免重复建设索引与权限模型。
3. 未来可能还需要 **连接器型同步**（云盘、在线文档）与 **多源融合**（本地 + 远端统一检索）。

当前仓库已具备部分积木：**工作区与文件路径解析**、`liteparse` 等解析链路、**会话消息 FTS**（`session_messages`）、**MCP 工具接入**、**元智能体记忆与工作区提示注入**。知识库是在此之上的 **专用 ingestion、向量/关键词混合索引、以及统一「检索 → 注入」契约**。

## 2. 能力模式（必须兼容 + 建议扩展）

### 2.1 模式 A：本地知识库（必选）

| 要素 | 说明 |
|------|------|
| **入口** | 工作区拖拽、文件夹挂载、或 KB 专用面板「添加资料」（与现有工作区 UX 对齐，避免两套路径长期分裂）。 |
| **解析** | 统一走可插拔解析管线：文本 / Markdown / PDF / Office / 图片 OCR（按权限与依赖可选）；失败时降级为可读错误与部分索引。 |
| **入库** | 切块（chunk）+ 元数据（来源路径、修改时间、分身份/分会话作用域策略需产品定调）；**向量索引** + 可选 **BM25/FTS** 做混合检索。 |
| **存储** | 默认本机（如 `~/.agenticx/` 下专用子目录）；密钥与路径不落日志。 |
| **问答** | RAG：检索 Top-K → 注入对话（或封装为 `knowledge_search` 类工具供模型显式调用）；引用须可追溯（片段 ↔ 源文件）。 |

**验收（产品级）：** 用户能在离线或仅本地模型场景下完成「加资料 → 问问题 → 答案带引用」闭环。

### 2.2 模式 B：远端知识库对接（必选）

| 子模式 | 说明 |
|--------|------|
| **B1：MCP** | 远端以 MCP Server 暴露检索/读文档能力（如 `search`、`get_chunk`、`list_sources`）。与现网 MCP 连接方式一致，便于第三方与自研 KB 服务接入。 |
| **B2：HTTP / OpenAPI 适配器** | 非 MCP 的云厂商 KB、内网网关，通过 **适配器** 映射到统一「检索结果」结构；前端与编排层不感知厂商差异。 |
| **B3：AgenticX 自有远端服务（可选产品）** | 若未来提供托管 KB，协议上仍应 **优先 MCP 或 OpenAPI**，与 B1/B2 同一抽象。 |

**验收：** 至少打通一条 MCP 样例与一条 HTTP 适配样例（可为 mock）；会话中能选择「仅本地 / 仅远端某连接 / 混合」。

### 2.3 模式 C 及其他（建议纳入路线图，不必首期全做）

| 模式 | 价值 |
|------|------|
| **连接器同步** | Notion / 飞书文档 / S3 / 网盘等定期拉取与增量更新。 |
| **会话内临时 KB** | 强依赖现有 `@file`、当前会话附件；与「长期 KB」区分，避免用户混淆。 |
| **多源混合检索** | 本地 + 多 MCP 并行或级联；需 **去重、配额、超时与失败部分成功** 策略。 |
| **团队/空间（远期）** | 多用户、共享索引与权限；与纯本机离线模型冲突时需明确 SKU 与架构分叉。 |

## 3. 与现有系统对齐（实现时约束）

- **身份与会话：** KB 作用域建议支持：全局 / 按分身 / 按会话（最小实现可先做全局 + 工作区根，再扩展）。
- **工具与提示词：** 检索结果进入 `meta_agent` 或会话层时，须遵守现有「上下文块」规范，避免仅报数量不报内容。
- **Desktop：** 乐观 UI、工作区面板、设置持久化等既有交互规范延续；大文件 ingestion 需后台任务与进度可感知。
- **隐私：** 本地索引默认不出站；远端检索须显式连接与权限说明。

## 4. 分期路线图（建议）

### 阶段 0：范围与契约（约 1 周）

**产出：** 统一数据契约（`RetrievalHit`：文本、分数、来源 URI、截断策略）、错误码、配额草案。

**任务：** 盘点可复用模块（解析、`session` FTS、MCP hub）；确定首期 **作用域模型**（全局 vs 分身）。

**验收：** 评审通过《API/契约说明》与「首期不做」列表。

### 阶段 1：本地 KB MVP（核心，视人力 3–8 周）

**内容：** 文件入库管线、chunk、embedding、本地向量存储、混合检索 API、`agx serve` 侧路由；Machi 最小 UI（资料列表、重建索引、问答时引用）。

**验收：** 离线可用；千级文档量级下检索延迟与内存占用在可接受范围（具体 SLO 另定）。

### 阶段 2：远端 MCP + 适配器（约 2–5 周）

**内容：** MCP 检索工具注册与配置；HTTP 适配器接口；设置页「知识库连接」与连接健康检查。

**验收：** 文档化一种官方示例 MCP；一种 mock HTTP KB 可走通。

### 阶段 3：体验与扩展（持续）

混合检索策略、连接器、团队版、与自动化/定时任务等联动（按需立项）。

## 5. 风险与开放问题

| 项目 | 说明 |
|------|------|
| **Embedding 与模型依赖** | 本机 embedding 模型体积与 CPU/GPU；是否允许调用云端 embedding 需产品与安全确认。 |
| **Web 端 Machi** | 若推进 Web 版，本地向量库与文件系统能力需降级方案（仅远端 KB 或浏览器侧受限存储）。 |
| **与 FTS 关系** | 会话 FTS 面向「聊天记录」；KB 面向「文档块」，二者可并存，避免混表导致产品语义混乱。 |

---

## 6. 需求条目（便于拆 issue）

**FR-1：** 用户可将本地文件/目录纳入知识库并完成解析与索引。  
**FR-2：** 用户可通过 MCP 或配置的 HTTP 适配器连接远端知识库并用于问答。  
**FR-3：** 问答结果应能展示可追溯引用（路径或远端资源标识）。  
**NFR-1：** 本地索引默认仅存本机；远端访问需用户可感知的连接状态。  
**NFR-2：** 大文件 ingestion 不应阻塞主聊天线程（异步与进度）。
